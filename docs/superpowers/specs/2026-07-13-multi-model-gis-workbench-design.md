# PyGeoModel 多模型 GIS 作业台设计规格

## 1. 背景

PyGeoModel 后端已经提供雷达探测、无人机侦察、观察所探测、炮兵覆盖、侦察车探测、机动通行性和空中航路规划七类模型。除雷达外，其余模型尚无前端入口。现有前端以雷达流程为中心，`App.vue` 同时承担任务状态、轮询、地图生命周期、图层、剖面、融合、DEM 和结果加载等职责，难以直接扩展到其他模型。

本次改造将前端建设为统一的多模型 GIS 作业台。用户在同一地图上下文中切换模型、配置参数、通过地图输入空间对象、提交计算任务、查看指标和管理结果图层。

## 2. 目标

- 为七类模型提供完整、可执行的前端工作流。
- 复用任务创建、轮询、结果获取、历史记录、删除和下载逻辑。
- 让地图成为主要工作区域，参数和结果分别位于稳定的左右工作面板。
- 保留雷达已有的 DEM、二维覆盖、三维体素、裁切体、剖面和融合能力。
- 将模型差异限制在清晰的模型定义、专属表单和图层适配器中。
- 保持现有 Vue 3、TypeScript、Element Plus、MapLibre GL 和 Three.js 技术栈。
- 支持本地直接启动，不依赖 Docker。

## 3. 非目标

- 不修改后端模型算法、任务存储方式或 API 路径。
- 不实现后端动态下发 JSON Schema 的表单系统。
- 不在首版增加用户账号、权限、跨设备任务同步或数据库持久化。
- 不在首版提供任意 GeoJSON 编辑器；只提供各模型所需的点、折线和威胁点编辑能力。
- 不为七个模型复制七套独立页面和任务状态实现。

## 4. 信息架构与布局

桌面端采用固定顶栏、模型导航、参数面板、地图画布和结果面板组成的作业台布局：

```text
+-----------------------------------------------------------------------+
| PyGeoModel | 当前 DEM / 数据状态                  任务历史 | 全局设置 |
+--------+------------------+---------------------------+----------------+
| 模型   | 参数配置          |                           | 任务与结果      |
| 导航   |                  |         主地图             |                |
|        | 空间输入          |                           | 状态与进度      |
| 雷达   | 模型参数          |                           | 核心指标        |
| 无人机 | 高级参数          |                           | 输出文件        |
| 观察所 |                  |                           | 图层控制        |
| 炮兵   | [开始计算]        |                           | 警告与诊断      |
| 侦察车 |                  |                           |                |
| 通行性 |                  |                           |                |
| 航路   |                  |                           |                |
+--------+------------------+---------------------------+----------------+
```

- 顶栏高度固定，显示产品名、当前 DEM、后端连接状态、任务历史入口和全局设置。
- 模型导航使用图标加短标签，宽度固定；切换模型不销毁其他模型的任务记录。
- 参数面板宽度约 320px，可折叠，按“空间输入、模型参数、高级参数”分组。
- 地图占据剩余空间，并承载选点、绘制路线、编辑威胁点、缩放至结果和图层反馈。
- 结果面板宽度约 360px，可折叠，使用“任务、指标、图层、文件”页签，避免纵向堆叠所有信息。
- 不使用浮动卡片包裹整段页面。面板通过边界线和背景层级分区，卡片只用于任务历史等重复项。
- 窄屏下模型导航收为图标栏，左右面板变为互斥抽屉；地图始终保留主要可视面积。

## 5. 视觉规范

- 整体采用中性浅灰工作区、白色工具面板和深炭色顶栏。
- 主操作和选中态使用蓝色；成功、等待、警告、失败分别使用绿、灰、橙、红。
- 不使用渐变、装饰光斑、超大标题或多层嵌套卡片。
- 面板和控件圆角控制在 4px 至 6px，主要分区使用 1px 分隔线。
- 正文字号以 13px 至 14px 为主，面板标题使用 15px 至 16px；不随视口缩放字号。
- 表单标签、数值和单位保持稳定对齐。数值参数优先使用带步进的输入框，枚举使用选择器或分段控件，布尔参数使用开关。
- 地图工具使用 Element Plus 图标或已有图标库中的熟悉符号，未知图标必须提供 tooltip。
- 图层项包含颜色色块、显示开关、透明度和定位按钮；图层颜色在不同模型间保持语义稳定。

## 6. 前端架构

### 6.1 应用壳层

`App.vue` 只负责组合页面骨架和初始化全局服务，不再直接实现模型业务。主要组件如下：

- `AppShell`：顶栏、导航、左右面板和响应式布局。
- `ModelNavigation`：模型分组、当前模型和模型切换。
- `ModelParameterPanel`：承载通用字段与模型专属表单。
- `MapWorkspace`：MapLibre 实例、地图工具、输入草图和结果图层。
- `TaskResultPanel`：任务状态、指标、输出文件、警告和图层控制。
- `TaskHistoryDrawer`：按模型筛选历史任务、恢复请求、定位结果和删除任务。
- `DemManager`：DEM 选择、上传、元数据和删除。

### 6.2 模型注册表

使用显式的 TypeScript 模型注册表描述稳定差异，不使用任意字符串驱动整个 UI。每个模型定义至少包含：

```ts
interface ModelDefinition<Request, Metrics> {
  id: ModelId;
  label: string;
  taskBasePath: string;
  createDefaultRequest(): Request;
  validate(request: Request): ValidationIssue[];
  metricDefinitions: MetricDefinition<Metrics>[];
  outputLayerDefinitions: OutputLayerDefinition[];
  spatialInput: SpatialInputDefinition;
}
```

注册表负责标签、默认值、接口基础路径、校验、指标格式和输出图层元数据。复杂联动不塞入通用字段描述：无人机与侦察车路线、通行性起终点、航路威胁点和雷达高级分析使用专属 Vue 组件。

### 6.3 API 与任务内核

将现有雷达专用 `client.ts` 拆分为：

- `api/http.ts`：API 基址、响应解析、错误归一化和资源 URL 解析。
- `api/dem.ts`：DEM API。
- `api/tasks.ts`：通用任务生命周期客户端。
- `api/radar.ts`：雷达剖面、融合等专属 API。
- `models/<model>/types.ts`：各模型请求、指标、输出和任务类型。
- `models/<model>/definition.ts`：模型定义、默认值、指标和图层描述。

通用任务客户端按模型定义的 `taskBasePath` 调用：

- `GET basePath`：任务列表。
- `POST basePath`：创建任务。
- `GET basePath/:taskId`：获取任务状态。
- `GET basePath/:taskId/metrics`：获取指标。
- `GET basePath/:taskId/outputs`：获取输出文件。
- `DELETE basePath/:taskId`：删除任务。

雷达仍通过 `/api/radar/coverage/:taskId/profile` 和 `/api/radar/fusion` 使用专属客户端。

### 6.4 状态边界

首版不引入新的全局状态库，使用 Vue composable 管理状态：

- `useModelWorkspace`：当前模型、每个模型的草稿和面板状态。
- `useTaskManager`：按 `modelId + taskId` 管理任务、轮询和历史记录。
- `useMapWorkspace`：地图实例、空间输入、图层实例和相机操作。
- `useDemManager`：DEM 列表、当前 DEM 和上传状态。

每个 composable 暴露明确的只读状态和命令。模型切换保留草稿与已完成图层，但默认只显示当前模型的图层。任务轮询独立于当前模型，切换页面不会停止正在执行的任务。

## 7. 模型适配

| 模型 | API 基础路径 | 空间输入 | 关键参数 | 主要结果图层 |
| --- | --- | --- | --- | --- |
| 雷达探测 | `/api/radar/coverage` | 雷达点 | 目标高度、距离、扫描模式、方位、波束和高级体素参数 | 可见区、遮挡区、理论范围、最低可见高度、体素和裁切体 |
| 无人机侦察 | `/api/uav/recon` | 单点或航迹 | 高度模式、姿态、传感器、视场角、距离和地形遮挡 | 足迹、可见区、遮挡区 |
| 观察所探测 | `/api/watchpost/detection` | 观察点 | 观察高度、目标高度、距离、扫描模式、方位和视角 | 可见区、遮挡区、理论范围 |
| 炮兵覆盖 | `/api/artillery/coverage` | 炮兵阵地点 | 高度模式、射程、方向射界、初速、射角、弹药和地形遮蔽 | 理论区、可达区、地形遮蔽区、采样点 |
| 侦察车探测 | `/api/recon-vehicle/coverage` | 单点或路线 | 车体朝向、桅杆、传感器、距离、扫描方式和目标高度 | 足迹、可见区、遮挡区 |
| 机动通行性 | `/api/mobility/accessibility` | 起点和终点 | 轮式/履带启用、速度、坡度阈值、道路倍率和搜索设置 | 轮式路径、履带路径、道路掩膜 |
| 空中航路 | `/api/air-corridor/planning` | 起点、终点、威胁点集 | 飞行器性能、高度层、走廊宽度、风险权重和威胁参数 | 航路、走廊缓冲区、威胁区、风险采样点 |

空间输入控件遵循同一交互约定：激活工具后地图进入明确的编辑状态；完成、撤销、清空使用固定位置的图标按钮；经纬度字段与地图标记双向同步；输入不完整时不允许提交并定位到缺失项。

## 8. 任务与数据流

1. 用户选择模型和 DEM，系统恢复该模型最近的未提交草稿。
2. 用户通过表单或地图完成空间输入，前端执行同步校验。
3. 提交后创建本地任务条目并调用对应 `POST` 接口。
4. 返回任务 ID 后，任务管理器按固定间隔轮询状态。进行中任务继续保留在历史列表中。
5. 任务完成后并行获取指标和输出清单；可视化输出按模型定义加载为地图图层。
6. 地图自动缩放到首个主要结果，但不覆盖用户随后进行的地图操作。
7. 用户可切换图层、查看指标、下载原始输出、恢复请求或删除任务。
8. 删除任务前明确提示会删除后端任务记录和输出目录；删除成功后再移除前端图层和历史项。

轮询使用单任务单计时器并在任务结束后释放。连续网络错误采用有上限的退避并显示“连接中断”；恢复连接后自动继续。模型切换、面板折叠或浏览历史不会重复创建轮询器。

## 9. 雷达能力迁移

雷达现有能力不降级，但重新归位：

- 基础覆盖参数进入通用参数面板。
- 可见、遮挡、范围和高度层进入统一图层页签。
- 剖面分析作为雷达地图工具，选择目标点后在右侧结果面板显示剖面。
- 多任务融合从任务历史中多选后启动，结果作为独立分析图层组。
- Three.js 雷达体、体素和裁切体保留现有渲染模块，通过雷达图层适配器接入统一图层生命周期。
- 恢复历史请求只修改雷达草稿，不自动重新提交任务。

## 10. 错误、空状态与并发行为

- 表单错误显示在具体字段附近，顶部只显示简短汇总。
- API 错误统一解析后显示用户可理解的信息，同时保留后端 message 供详情展开查看。
- 任务失败时保留请求、进度、警告和已存在的输出文件，允许恢复参数重试。
- 输出文件缺失或 GeoJSON 加载失败只标记对应图层失败，不将整个已完成任务改为失败。
- 未选择 DEM、任务为空、指标未生成和输出为空分别使用独立空状态。
- 同一模型允许多个并发任务；每个模型只维护一个未提交草稿。
- 删除当前正在显示的任务时，先清除其图层，再选择该模型最新的剩余任务。
- 后端不可达时禁用提交和上传，但允许浏览已经加载到内存中的任务结果。

## 11. 可访问性与响应式

- 所有图标按钮具有可读的 tooltip 和 `aria-label`。
- 键盘可以访问模型导航、页签、表单和任务历史；焦点样式清晰可见。
- 状态不能只依赖颜色，必须同时使用图标或文字。
- 面板滚动区域彼此独立，固定操作区不会遮挡表单内容。
- 桌面宽度低于约 1100px 时收窄导航；低于约 800px 时左右面板改为抽屉并互斥打开。

## 12. 测试与验收

### 12.1 自动化测试

- 为模型注册表增加单元测试，确认七个模型 ID、API 路径、默认请求和图层 kind 完整。
- 为通用任务客户端测试 URL 拼接、创建、轮询结果归一化、错误解析和删除。
- 为任务管理器使用假计时器测试完成、失败、网络重试、切换模型和计时器清理。
- 为各模型请求校验测试关键约束，例如最小距离小于最大距离、路线至少两个点、至少启用一种车辆、航路起终点完整。
- 为通用空间输入和任务结果组件增加关键交互测试。
- 使用 `vue-tsc` 和 Vite 生产构建作为必过检查。

项目当前没有前端测试框架，实施时引入 Vitest 和 Vue Test Utils；不引入完整端到端测试平台作为首版的硬依赖。

### 12.2 浏览器验收

- 在桌面和窄屏视口检查无面板重叠、文字溢出和地图空白。
- 逐个模型完成一次表单输入、地图输入和请求提交。
- 对可成功运行的任务检查指标、输出下载、图层显隐和缩放定位。
- 对失败任务检查错误详情、恢复请求和重试流程。
- 检查模型切换后任务仍继续轮询，返回原模型后状态和图层可恢复。
- 检查雷达剖面、融合和三维图层没有功能回退。

涉及 GDAL viewshed 的模型依赖本机 `gdal_viewshed`。本地环境未配置该命令时，相关任务失败属于环境限制，但前端必须正确呈现失败状态；不以伪造成功结果替代验收。

## 13. 实施边界与迁移顺序

实施按共享基础、二维模型、路线模型、雷达高级能力和视觉验收分阶段完成：

1. 建立应用壳层、模型注册表、通用 API、任务管理和 DEM 管理。
2. 接入观察所、炮兵等单点二维模型，验证通用流程。
3. 接入无人机、侦察车、机动通行性和空中航路的路线及多点编辑。
4. 将雷达现有二维、剖面、融合和三维能力迁移到新架构。
5. 完成响应式、视觉统一、自动化测试和七模型浏览器验收。

迁移期间不保留新旧两套永久入口。功能在新壳层中达到等价后删除对应旧编排代码，现有底层地图渲染模块在满足统一生命周期接口后继续复用。

## 14. 完成标准

- 七个模型都可从导航进入，填写或绘制必需输入并提交真实后端任务。
- 所有模型共享一致的任务状态、历史、指标、文件和删除体验。
- 每个模型的主要 GeoJSON 输出均可独立显示、隐藏、调整透明度和定位。
- 雷达现有二维、剖面、融合和三维能力可用。
- `App.vue` 不再承担模型实现细节，任务、地图、DEM 和模型适配职责边界清晰。
- 前端类型检查、生产构建和新增自动化测试通过。
- 桌面及窄屏布局无不可用的遮挡、溢出或地图空白。
- 项目继续通过本地前后端进程启动，不要求 Docker。
