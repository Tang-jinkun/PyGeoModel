# 独立模型演示场景数据设计

## 1. 背景

项目当前使用 `dem_20260713_080113_884937cf`（札达县 DEM），雷达探测已经存在可用于前端展示的完整任务与输出。除雷达外，工作台还包含无人机侦察、观察所探测、炮兵覆盖、侦察车探测、机动通行性和空中安全走廊六个模型，但缺少统一生成、真实计算和重复恢复的演示场景。

本设计为六个模型分别选择最能体现其计算能力的独立区域。场景之间不要求空间关联。所有结果必须由现有后端基于目标 DEM 真实计算，不能直接伪造任务状态、指标或输出文件。

## 2. 目标

1. 为六个非雷达模型各生成一份独立、确定性、可重复构建的合成输入场景。
2. 通过现有 API 提交真实任务，并保留为前端能够恢复和展示的历史任务。
3. 确保每个场景都有非空、可解释且适合演示的主要指标和地图图层。
4. 保存请求、任务 ID、输出清单和验收结果，支持清理任务后的重新生成。
5. 不修改或删除现有雷达任务，不直接写入后端任务存储目录。

## 3. 非目标

- 不生成新的 DEM 或修改札达县 DEM。
- 不要求六个模型构成同一个联合演练场景。
- 不伪造后端任务 JSON、指标 JSON 或计算输出 GeoJSON。
- 不把 DEM、任务目录和大体积计算结果提交到 Git。
- 不在本阶段调整前端布局、样式或模型计算算法。

## 4. 方案

采用可复现的场景生成器和任务运行器：

- 场景生成器读取 DEM 元数据和降采样地形，选择候选点、路线与辅助要素，并生成请求 JSON/GeoJSON。
- 任务运行器调用现有模型 API，轮询任务状态，读取指标和输出清单，并执行模型级验收。
- 后端仍是 `data/tasks` 和 `data/outputs` 的唯一写入者。
- 场景索引使用 DEM ID、场景版本和请求哈希识别已有成功结果。默认复用匹配任务，显式重建时才重新计算。

一次性手工提交 API 缺少可靠的恢复能力；直接写任务和输出目录会绕过存储契约。这两种方式均不采用。

## 5. 文件组织

```text
data/demo-scenarios/
  dem_20260713_080113_884937cf/
    scenario-index.json
    uav-recon.json
    watchpost-detection.json
    artillery-coverage.json
    recon-vehicle.json
    mobility-accessibility.json
    air-corridor.json
    road-network.geojson
    air-defense-threats.geojson

scripts/
  generate_demo_scenarios.py
  run_demo_scenarios.py
```

`data/demo-scenarios` 保存本机运行时场景和索引，不要求纳入 Git。两个脚本纳入版本控制。每个场景 JSON 使用包含 `scenario` 元数据和 `request` 对象的外层结构；`request` 必须是对应后端 API 的原生请求结构，运行器只提交该对象。辅助 GeoJSON 同时保持人可读和可在地图工具中检查。

## 6. 公共地形分析

场景生成器读取以下 DEM 元数据：

- CRS：`EPSG:4326`
- 范围：`[78.39572341, 30.449041933, 80.942515968, 32.70424882]`
- 原始尺寸：`9171 x 8121`
- 原始角分辨率：约 `0.0002777` 度

生成器对 DEM 降采样，计算有效像元、高程、坡度和局部地形起伏。候选几何必须位于有效区并与数据边界保持安全距离。生成使用固定随机种子；相同 DEM、场景版本和脚本版本必须产生相同请求。

生成器按确定顺序提供候选区域。若运行器发现某个候选场景未通过模型验收，它调用生成器选择该模型的下一个候选并重新提交，同时记录尝试次数。不得通过随机改变结果而破坏可复现性。

## 7. 模型场景

### 7.1 无人机侦察

- 在山谷与山脊交错区生成 6 至 10 个航点。
- 航迹长度目标为 10 至 20 km。
- 航点高度为 300 至 800 m AGL，航向根据相邻航段计算，并加入有限俯仰变化。
- 使用光电或热成像传感器，设置能覆盖航迹两侧地形但不会覆盖整个分析区的视场和距离。
- 场景应同时产生航迹、可见区、遮挡区和多航点重叠覆盖。

### 7.2 观察所探测

- 选择局部高程较高、周围仍有更高山脊的候选点。
- 使用指向谷地和对侧山体的扇区扫描。
- 探测距离目标为 8 至 15 km。
- 场景应产生理论范围、可见区和清晰的山后遮挡区，避免接近全可见或全遮挡的极端结果。

### 7.3 炮兵覆盖

- 选择坡度较缓且主要射界前方存在山体的阵地点。
- 射程目标为 2 至 12 km，方向射界为 90 至 140 度。
- 启用 DEM 高程和地形遮蔽分析，并生成足够的弹道采样点。
- 理论覆盖区、可达区、地形遮蔽区和采样点必须均非空。

### 7.4 侦察车探测

- 在低坡度谷地生成 5 至 8 个地面巡逻点。
- 路线长度目标为 5 至 12 km，车体朝向按路线切线计算。
- 使用桅杆光学或热成像设备，探测距离为 3 至 6 km。
- 沿途覆盖应连续，并形成由两侧山体造成的遮挡区和多点重叠区。

### 7.5 机动通行性

- 起点和终点由坡地或山脊分隔，但均位于有效且可搜索的地形上。
- 合成道路网包含主路、支路和便道三类 LineString，并使用道路等级属性映射速度倍率。
- 道路走向优先选择较平缓但更绕行的线路。
- 轮式车辆具有更高道路速度、较低最大坡度和更高越野惩罚；履带车辆速度较低，但允许更大坡度和越野捷径。
- 两种车辆必须均找到有效路线，并在路线形态、距离或耗时上产生可见差异。

### 7.6 空中安全走廊

- 起终点距离目标为 20 至 40 km。
- 在直线路径附近生成 3 个合成威胁点，作用距离、高度范围和威胁等级相互错开。
- 配置多个 AGL 高度层并允许高度改变。
- 规划结果应成功到达终点，并通过横向绕行或高度改变规避高风险区域。
- 航路、走廊缓冲区、威胁区和风险采样点必须均非空。

所有场景外层元数据必须包含 `"synthetic": true`，辅助 GeoJSON 的集合及要素属性也必须带有 `synthetic` 或 `demo` 标识。运行器提交 API 时不把这些标识混入原生请求对象，避免改变现有请求契约。合成数据不能被解释为真实业务或观测数据。

## 8. 执行流程

1. 检查后端健康接口、DEM 是否存在以及 DEM 元数据是否与场景版本匹配。
2. 运行场景生成器，输出六份请求和辅助 GeoJSON。
3. 对所有请求进行本地结构、范围、距离和参数校验。
4. 任务运行器按固定模型顺序串行提交任务，避免多个 DEM 重计算同时占用内存。
5. 按固定间隔轮询任务，直到 `finished`、`failed` 或模型级超时。
6. 对已完成任务读取指标和输出清单，验证必需图层内容。
7. 验收失败且仍有候选次数时，仅为该模型生成下一个确定性候选并重复步骤 3 至 6。
8. 将最终任务结果和验收结论写入 `scenario-index.json`。

一个模型失败不阻断后续模型。运行器最终返回非零退出码并汇总所有失败场景，便于修复后只重跑失败项。

## 9. 错误处理

- 网络超时和临时 `5xx` 使用有限次数、带退避的重试。
- API 参数校验错误立即停止该场景，保留请求和响应，不进行无意义重试。
- 坐标位于无效地形、路线不可达或关键结果为空时，选择下一个确定性候选区域并重新生成该场景。
- 达到候选上限或任务超时时，记录错误、请求、任务 ID、轮询时长和尝试次数。
- 重跑默认复用请求哈希匹配且验收成功的任务；`--rebuild` 显式创建新任务。
- 脚本不自动删除旧任务，也不修改现有雷达历史。

## 10. 场景索引

`scenario-index.json` 对每个模型记录：

- 场景名称、版本和 `synthetic` 标识。
- DEM ID、请求文件和原生 `request` 对象的规范化哈希。
- 任务 ID、最终状态、耗时和重试次数。
- 核心指标摘要。
- 输出文件及其 kind、文件名和媒体类型。
- 验收是否通过以及失败原因。

索引写入采用临时文件后原子替换，避免进程中断留下半份 JSON。

## 11. 验收标准

- 六个请求均通过后端模型校验并创建真实任务。
- 六个任务最终状态均为 `finished`，并能由前端历史任务列表读取。
- 无人机具有航迹、可见区和遮挡区，航迹长度大于零。
- 观察所具有理论范围、可见区和遮挡区，遮挡率不是全零或全一。
- 炮兵具有理论区、可达区、遮蔽区和采样点。
- 侦察车具有路线、可见区和遮挡区，覆盖点不少于两个。
- 机动通行的轮式与履带式路线均有效，且距离或耗时存在差异。
- 空中走廊成功找到航路，并具有威胁区、航路、缓冲区和风险采样点。
- `scenario-index.json` 完整记录六个通过验收的场景。
- 重新运行且不使用 `--rebuild` 时，不创建重复任务。

## 12. 测试

- 单元测试覆盖 DEM 候选筛选、固定种子确定性、路线点顺序、边界余量和请求哈希。
- 为六种请求分别测试结构校验和坐标有效性。
- 使用模拟 API 测试任务提交、轮询、瞬时错误重试、失败隔离、超时和索引原子写入。
- 使用现有 Docker 后端执行一次完整集成运行，验证六个任务和所有必需输出。
- 在前端逐个打开六个历史任务，检查指标、图层加载、显隐、定位和下载。
